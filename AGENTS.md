## Guidance

Every code change MUST include a documentation pass before the task is considered complete. After making any code change, identify which docs could be affected (AGENTS.md, README.md, ARCHITECTURE.md, and the relevant docs/ topic files) and update them in the same change so the docs do not drift from reality. If the change is genuinely doc-irrelevant (a typo fix, a pure dependency bump, a no-op refactor), say so explicitly rather than silently skipping the pass. Treat updated docs as part of the deliverable — do not defer them to "later".

Consider writing an ADR whenever you are planning a multi-step or large-blast-radius architectural change — anything spanning multiple PRs or phases, anything that establishes a pattern or forecloses an alternative, anything touching the simulation engine, persistence shape, concurrency, payments, or deploy. Bug fixes, dependency bumps, local refactors, and features that fit cleanly inside an existing pattern do not need an ADR. When unsure, propose one in the planning phase rather than after the fact.

ADRs are immutable once accepted. Treat them like database migrations: never rewrite the body of an accepted ADR to reflect a later decision. When a decision evolves, capture it in a new ADR rather than editing the old one — both supersession and refinement keep the spirit of immutability. If the new ADR reverses or replaces the prior decision, it **supersedes** it (`Supersedes ADR-NNNN` in its Status line, and the prior ADR's Status line gains a `Superseded by ADR-MMMM` pointer). If the prior decision still stands but the new ADR extends or adjusts its rationale — or records newly-discovered detail — it **refines** it (`Addendum to` / `Refines ADR-NNNN` in its Status line, and the prior ADR's Status line may gain a brief `Refined by ADR-MMMM` pointer). In both cases the only change to the prior ADR is a short forward-pointer on its Status line; the decision and reasoning in its body are never rewritten. Typo fixes, dead-link repairs, and markdownlint corrections are not revisions and are fine.

Keep docs/adrs/README.md in sync. Every new ADR file must be added to the index in the same change. Every ADR file (and the index) must pass markdownlint with zero errors — run `just lint-docs` (`just lint-docs --fix` auto-corrects the mechanical rules). Cite PRs and issues with full URLs, not bare numbers.

Built-in tools follow a construction convention. Every tool struct pairs `#[derive(Default)]` with an explicit `pub fn new() -> Self` and is registered as `Arc::new(XTool::new())` in `ToolRegistry::with_builtins` — even a stateless unit-struct tool. The `new()` is a forward-compatible seam, not boilerplate: it lets a tool gain configuration later without churning any construction site (see `ExecTool`, which grew an `ExecConfig` and a `with_config()` while every caller kept using `new()`). Match this when adding a tool rather than registering the bare struct literal.

CI runs through `just`. Every GitHub Actions job invokes a `just` target rather than raw cargo/npx/tooling, so the CI environment stays isomorphic with local dev — what passes under `just ci` locally passes in CI. Toolchain setup steps (Rust, Node, just itself) are the only non-`just` glue, since they must exist before `just` can run. Split CI into separate jobs by concern — e.g. docs lint (`just lint-docs`), and the Rust gate split into a runtime job (`just runtime-ci`, whose NATS-backed tests spawn their own broker from the pinned `nats-server`, plus Node for the MCP tests) plus hermetic store and dashboard jobs (`just store-ci`, `just dashboard-ci`) so one suite's failure never masks the others — each calling its own target. `just ci` invokes those same targets rather than re-implementing them, which is what keeps the local gate and CI from drifting: adding a suite to CI means adding its target to `just ci`, and nothing else.
