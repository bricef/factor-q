# factor-q — an in-depth review

*Reviewed 2026-07-09 against `main` @ `973e822` (352 commits, 2026-03-12 → 2026-07-09; ~49.5k lines of Rust across `fq-runtime` and `fq-store`; 29 accepted + 3 draft ADRs; 17 closed and 6 active plan documents; 62 GitHub issues/PRs, all opened since 2026-06-30).*

*Method note: full clone, all top-level docs, all ADR titles plus deep reads of 0006/0007/0014/0016/0026/0027/0028, both self-assessments, the M0 and graph-executor plans, the worker-ergonomics and orchestration aspirational docs, the event schema, reducer/oracle/sim/tool source, the Go watcher, and the complete issue list. I could not compile the workspace in this sandbox (edition 2024 needs a toolchain I can't fetch here) and CI check-runs aren't visible unauthenticated, so correctness claims below are a static read of the code and test machinery, not an executed verification.*

---

## 1. Approach

The thesis is coherent and genuinely differentiated: not another interactive coding harness, but a continuously running, event-driven runtime where the human is one trigger source among many. Every load-bearing early decision is consistent with that thesis rather than merely compatible with it — headless-first is actually headless (the CLI is a NATS/projection client, the daemon outlasts sessions), cost controls landed before the first incident rather than after (ADR-0004, enforced post-call and re-checked on resume), and the event bus is the spine in fact, not just in the README.

The single strongest architectural idea is the reducer principle (ADR-0014) and its upward extension in ADR-0007: the agent expresses intent, the harness owns all I/O, and the boundary is made structural by the three-layer event format. This one shape is doing an enormous amount of work. Suspension, replay, crash recovery, graceful-drain deploys (ADR-0027 — deploys as suspend-at-step-boundary rather than kill-and-recover is a genuinely elegant consequence), hermetic testability with zero LLM calls, and now the multi-agent extension all fall out of it rather than being separately wired features. ADR-0007's reframing — "the messaging menu is the wrong altitude, because agents do not communicate over the transport; the harness does" — is the kind of move that collapses a design space rather than picking within it. The follow-on decisions (one graph executor, spawn as desugared sugar, per-traversal budget with an ε floor making budget a *complete* termination guarantee for cyclic graphs) are all downstream of that reframe and all correct in my judgment.

The second through-line, safe-by-construction (principle 3), shows a visible escalation of commitment: phase-1 path allowlists → typed operations instead of free-form APIs (ADR-0016) → tool-scoped isolation with a harness-owned VFS (ADR-0028/0029). What earns trust here is the honesty about where the current implementation falls short of the principle — the shell tool's module doc enumerates its own gaps (no PATH restriction, no network isolation, no rlimits, no seccomp) and the two ugliest gaps are open tickets in plain sight (#34: env allowlist parsed but not plumbed; #35: network allowlist declared but never enforced). Most projects bury this; here it's the top of the file.

The Q200 north star is memorable and the fusion naming is apt, but until 2026-07-05 it was decoration — nothing measured it and M0 had no plan. The project assessment called this out in unusually blunt terms and the correction was immediate and substantive. The M0 plan's measurement design is the best part: five proxies read as pairs and trends, explicitly never composited into a score; a stated "its lie" column for each proxy; the trivial-work-drift failure mode named and guarded qualitatively; baseline as an ongoing expert+frontier *distribution* rather than a one-off anchor; throughput deliberately excluded from the Q plot as a category error. This is more measurement epistemics than most funded teams apply to their KPIs.

One tension worth naming because it will recur: the project's gravity is platform-ward. Within four days of committing to loop-first, two new deep-infrastructure fronts opened (ADR-0028's VFS model and the ADR-0029 FUSE bake-off). Both are defensible — tool-scoped isolation is arguably the prerequisite for letting an agent edit factor-q's own source — but the pattern the assessment diagnosed ("pillar 2 grew from a scope item into a platform") is a standing temptation, and the FUSE work is exactly its shape. The discipline that will matter is not avoiding platform work but forcing each piece to justify itself against what the running loop needs *next*.

## 2. Rigour

This is the standout dimension, and I want to be precise about why rather than just say "high."

The verification culture is real and layered. The trace oracle (`test_support/oracle.rs`) states the canonical event sequence as an explicit grammar automaton over captured events — including the subtle cases (nested server-initiated sampling triples inside tool spans, degenerate synthetic-error spans, exactly-one-terminal, parent-pointer monotonicity) — and collects all violations rather than failing fast. The sim world swaps every platform dependency for a deterministic double (scripted LLM, recording tool, in-memory sink with publish-fault injection, counter clock, tempdir WAL) so that crash points sit between operations exactly as the recovery semantics define them, and the same seed reproduces the same trace. Resume-equivalence, crash DST, budget properties, and a scalable soak sit on top. The claim that this machinery found seven real bugs is credible from the commit trail: "ack triggers after durable start (close the ack→first-WAL-write window)" and "NAK transient pre-WAL failures instead of blind-ACK" are precisely the at-least-once seams that bite event-sourced systems in production, being found and closed pre-production.

The TLA⁺ work on GC deserves specific credit because it's used where it earns its keep rather than as decoration: safety across 226k states, liveness under an explicitly reasoned fairness split (strong fairness for the collector's audit, weak for writers — with the note that TLC showed weak-alone lets a crash-orphaned generation leak forever), a bounded-crash counter so liveness holds modulo finite faults, and — the part that impressed me most — an *independent* Python checker that disagreed productively, finding the Materialize gap the TLA⁺ model's abstraction had elided. Cross-checking your formal model with a second oracle is the move that separates people who use TLA⁺ from people who understand what it does and doesn't buy.

ADR discipline is likewise real: immutability-with-supersession treated like database migrations (AGENTS.md spells out supersedes vs refines semantics, and the index carries the forward pointers — ADR-0011's persistence half is marked partially superseded by 0026 while its bus role stands, which is exactly the right granularity). ADR-0007 is a model of the form: it names the assessment critique it answers, reframes, decides eight numbered things, derives consequences, sequences the build (declared two-node graph first, testable with zero LLM calls), and defers named open questions with the sharpest one flagged as such.

The rarest habit of all: the 2026-07-05 project assessment is a deliberately-honest self-review, and within four days its recommendations were *acted on* — STATUS.md exists and is good, the M0 plan exists and answers §1 directly, ADR-0007 graduated answering §7, the dogfood loop went live, aspirational docs got labeled. A critique loop that closes is worth more than any individual artifact in the repo.

Where rigour is thinner, mostly by the project's own admission: the assessment's §3 point — rigour concentrates where verification is easiest — still partially stands. Storage got TLA⁺; the reducer got its oracle arc; but context-window management, the thing that actually kills long-running autonomous agents, remains not just unbuilt but *undesigned* (an aspirational doc exists; nothing in committed/, no plan, no tickets). And structurally, `runner.rs` at 5,550 lines is a concentration risk — well-tested, but the verification net is doing load-bearing work that module boundaries would normally share; `bus.rs`'s mixed responsibilities are already self-ticketed (#43).

## 3. Correctness

With the caveat that I read rather than ran: the correctness *machinery* is convincing, and the correctness *thinking* visible in fixes is the right kind. Flaky tests are treated as bugs with causes, not retried into silence — #18 (single clock read for the terminal-state upsert) and #25 (EPIPE fixed by graceful MCP stdio teardown rather than a retry band-aid) both diagnose the race instead of papering it. 634 test annotations across the workspace, tiered (hermetic vs NATS-gated vs the real-LLM smoke suite), with CI driven entirely through `just` targets so local and CI are isomorphic — a small decision that quietly eliminates a whole class of "works on my machine."

Sharp edges I'd rank by bite, all of which the project already knows about, which itself is the finding:

The most conceptually dangerous is the sandbox schema over-promising (#34/#35). An agent definition can declare `sandbox.env` and `sandbox.network` today and the runtime silently honours neither. That inverts the project's own principle 3 — the failure mode of an unenforced declared boundary is a *silent success with wider-than-intended reach*, the exact thing the allowlist philosophy exists to prevent. I'd argue these two tickets are miscategorised as tech-debt; until enforced, the honest fix is to make declaring them a validation error (fail the definition, per principle 7's "respect known constraints") so the schema cannot promise what the runtime doesn't hold. That's an afternoon and it converts a trust hazard into a loud gap.

Second, the trigger consumer has no `max_deliver` and no dead-letter (#49) — a poison trigger NAKs and redelivers unbounded. In a runtime whose whole identity is unattended operation, that's the classic 3 a.m. failure. Third, pricing coverage is being closed in-flight (#61/#62, fail-fast at startup) — right fix, since budget enforcement is only as sound as the pricing table under it.

One correctness-of-story item: the ADR-0026 decision (CAS-backed archive as the event log's system of record, NATS demoted to transport) resolves the assessment's §5 two-sources-of-truth critique on paper, and the archive hand-off covers invocation payloads — but whether the full *event trail* now outlives the 30-day JetStream retention in practice is something I couldn't confirm from a static read. If any consumer still assumes replay-from-NATS beyond retention, that's the residual.

## 4. Consistency

For a repo carrying this documentation volume, drift control is genuinely good: markdownlint and a link checker in CI, the ADR index updated in the same change as every ADR (enforced by convention in AGENTS.md), STATUS.md with the epistemically correct footer ("if this contradicts git log, trust the log and fix this file"), and the committed/aspirational/plans-active/plans-closed taxonomy doing exactly what the assessment asked of it.

The drift I did find is small but worth fixing because the front door is where it lives. The M0 plan lists "triggers, and cron" among building blocks that already exist, while STATUS.md correctly lists scheduled triggers as unbuilt and no scheduler exists in the runtime source — presumably the daily doc-drift run is host crontab calling `fq trigger`, which is fine, but the plan should say so rather than implying a runtime capability. The README leads with the `curl | sh` installer as the primary install path while STATUS and the fq-store README both note it awaits the first release — a newcomer's literal first command 404s, which is a poor first impression for a project this polished. The backlog's hot-reload entry still describes the file-watcher design although `fq reload` (control-message reload, refresh-between-invocations) shipped via PR #3 — the entry should be closed or rewritten as "watcher variant of the shipped command." And STATUS's operator-surface list predates `reload`/`drain`/`invocation`/`workers` — consistent with its update-at-milestones policy, but the list reads as exhaustive to a newcomer.

That this is the complete list of drift I could find, in ~90 markdown files under active daily development, is itself the consistency verdict.

## 5. Feasibility

Split the question the way the project itself does. As a *product for others*, the open surface is oceanic: API layer (ADR-0006 undecided), container/VFS isolation, memory, skills, compaction, observability floor, multi-agent executor, NATS auth, releases. No solo maintainer ships all of that to product quality, and the BSL licence plus zero-star early-days state means no cavalry is coming. But that's not the near-term goal, and judging it as one would be a category error.

As a *personal leverage instrument* — which is what the Q ladder actually defines — it's feasible, and the mechanism is already demonstrably working: the v0 dogfood loop's findings (prompt caching, SIGPIPE, broker separation) reordered the backlog, the watcher grew outcome-observation because a stranded issue (#15) demanded it, and the last ten days show the load pulling the roadmap exactly as the assessment prescribed. Velocity under the oracle discipline is strong — M1c in ~3 days, M2 in ~2 including a security review, drain from ADR to shipped in ~2, the FUSE bake-off from spec to scored ADR in 2. At that cadence the M0 loop is plausibly weeks away, not months — with two concrete blockers, one ticketed and one not.

The ticketed one is env plumbing (#34): the propose node needs `GH_TOKEN` in its child environment and today `env_clear()` drops everything not in a static empty allowlist. Known, small.

The unticketed one is the load-bearing risk of the whole M0 milestone: **there is no context management of any kind, and the agent can't even see the pressure.** A propose-node doing multi-file comprehension plus test-iterate-until-green accumulates context fast; today the invocation simply dies at the provider limit as a failure, wasting its entire spend. Interactive harnesses in this class live and die by compaction. Either M0 tasks must be scoped to fit comfortably in-window (viable, but it caps the "non-trivial" bar M0 sets for itself), or the null compaction policy needs its principle-6 reference implementation — even truncate-oldest-behind-a-summary — before the loop attempts real tasks. Pillar 5 being simultaneously unstarted, undesigned-in-committed, unticketed, *and* on the critical path of both M0 and Memory ("store important context before compaction") makes it the single most consequential gap in the project. Everything else on my list is an improvement; this one is a wall.

## 6. Project-management discipline

The history has two regimes and the transition is the interesting part. Through June, task management was plans-as-documents per ADR-0001 — and the closed-plans directory is a genuinely excellent artifact: seventeen dated plans with "decisions taken while building" sections tell the project's true history better than most changelogs. Then from 2026-06-30 GitHub issues appear and densify sharply from 07-07 — not process drift but a deliberate re-platforming of task intake so that *machines* can consume it: the Go watcher's label state machine (`ready → in-progress → in-review → done`, with relabel-before-trigger as the idempotency mechanism and bounded retry on transient failure) turns the issue tracker into the loop's work queue. Reshaping your own PM process to be agent-legible is principle 1 applied to management, and few projects think to do it.

Hygiene is strong: conventional commits with scopes and issue references, PR-per-change even solo (52 of 62 items closed — nothing festers), issue form templates, a small honest label set, and AGENTS.md making the docs-pass a hard requirement of every change. Gaps are minor: no GitHub milestones/projects (the roadmap lives in docs, fine, but plan↔ticket linkage is one-directional in places); labels only exist post-06-30 so the earlier history is only navigable via plans; and the known security posture lives partly in tickets (#35) where SECURITY.md — currently a stub — should carry it, the way STATUS already carries the NATS-no-auth caveat.

## 7. The implied roadmap

Synthesising docs plus tickets, three horizons emerge. **Near (days–weeks):** harden the M0 loop — workspace sync-before-run (#14), env plumbing (#34), watcher gh-CLI→API (#30), provider pluggability (#58 with the spike already merged, #45); land the two-node graph vertical (propose→review) as the first signature-typed traversal; pricing fail-fast (#61/#62, in flight). **Mid:** storage M3→M4→M5 held lean per the assessment, then Memory and Skills MVPs on top; the ADR-0028/0029 VFS build; the observability floor (JSON logs shipped, then stuck-invocation detection #37, metrics, `fq doctor` #50); dead-lettering (#49); trigger-boundary cost control (#42). **Later:** the API layer (self-identified as the long pole, still undecided), container-isolation residual, scheduled triggers, releases, NATS auth, multi-runtime scaling.

Two observations on the shape. First, the roadmap is now honestly dual-track — loop-first M0 pulling from one side, the storage platform completing from the other — with the graph executor as the hinge where "multi-agent runtime" stops being aspirational identity and becomes running code. The two-node vertical is exactly the right size for that hinge, and its deliberate exposure of the identity gap (a reviewer's verdict is only trustworthy if the reviewer is a distinct, grant-scoped identity) shows the plan is designed to *surface* the next problem, not just ship a feature. Second, as argued above, context management is the roadmap's ghost: named everywhere as a dependency, planned nowhere. I'd put a design date on it before the M0 loop's first non-trivial task, even if the design's conclusion is "truncation stopgap now, real compaction post-Memory."

One structural recommendation on the long pole. ADR-0006 frames the API decision as transport-first (REST vs gRPC vs WebSocket vs combo). Given ADR-0016 already commits the project to typed operations, and given the graph work is making signatures the primitive, the stronger move is to decide the **operation registry** first — every runtime capability as a typed operation with schema, permissions, and audit metadata — and *derive* the surfaces from it: REST+SSE for humans and webhooks, and **MCP server** as a first-class derived surface. factor-q has a full MCP *client* and no MCP *server* face; yet fq-as-an-MCP-server is the cheapest path to letting other harnesses (and factor-q's own agents, and frankly me) trigger, inspect, and query it — and MCP isn't even on ADR-0006's option list. Define once, derive the interfaces; it dissolves the transport debate rather than settling it, and it's the same pattern the rest of the architecture already follows.

## 8. From the agent's seat

Principle 1 declares LLMs first-class users and invites exactly this exercise, and the worker-side-ergonomics doc anticipates much of what I'd say — `SelfInspect`, checkpoints, `Consult`, structured `ToolError`, structured uncertainty are all well-argued and I co-sign the doc's own judgment that structured errors are the one *categorical* improvement in the set. So rather than restate it, here is what the cockpit actually feels like today, where I'd reprioritise the aspirational surfaces, and what's missing even from them.

**The cockpit today.** I wake with a system prompt (the agent body), a trigger payload as my first user message, up to four built-ins plus whatever MCP servers I've been granted, a budget enforced after every call, and an iteration ceiling. Several things about this are already better than most harnesses I could run in: `self_inspect` exists at all, so when asked about my budget or model I can *answer instead of confabulate* — the design principle behind it is one I'd want every runtime to adopt; nothing-by-default means my tool list *is* my authority, no guessing at ambient capability; sandbox violations come back as tool errors I can adapt to rather than hard kills; argv-only shell means I never fight quoting; and the full event trail means the runtime's account of what I did is one I'd trust — auditability is an agent-welfare feature as much as an operator one.

**The gaps, in the order they'd actually hurt me on an M0-class task:**

*I am blind to context pressure, and there is no parachute.* `SELF_INSPECT_SECTIONS` covers budget/iterations/model/tools but not `context` — the one section the aspirational doc includes that the implementation dropped, and the one whose absence is fatal because there's no compaction behind it either. My first signal of context exhaustion is death, after the operator has paid for the whole run. Before any compaction work: surface tokens-in-use and window size in `self_inspect`, and have the runner inject a soft warning at ~80% ("wrap up or summarise"). That's a read path over state the runner already holds, i.e. exactly the near-free exposure principle 2 promises.

*I have eyes but no map.* `file_read` exists; `file_list` does not. An agent granted only `fs_read` cannot discover a single path — it can read files it already knows and nothing else. Today the workaround is granting `shell` just to run `ls`/`grep`, which hands me arbitrary process execution to obtain what is morally part of *read*. A `file_list` (glob-shaped) and a `file_search` (grep-shaped, structured hits) built-in is simultaneously an ergonomics win and a least-privilege win — it lets you *narrow* my grants. For code loops specifically, a `workspace_status` read (git-status/diff-shaped) turns "what have I changed so far" from N shell round-trips into one typed call, and doubles as the natural payload for the review node and the correction-ratio metric.

*Errors are prose and every failed idiom costs a full round-trip.* Argv-only shell is the right call and I endorse the injection-safety reasoning — but models habitually reach for `| head` and `>`, and each failure burns an iteration plus tokens. Two mitigations that preserve construction: make the error message *teach* (when argv contains `|`/`>`, say "no shell layer; pass argv; output is auto-capped" rather than a bare exec failure — error messages are prompts), and consider a small typed `pipeline` tool (a list of argv stages the host wires with pipes itself — still no shell, still constructed). And promote `ToolError` from the aspirational doc soon: `retryable`/`retry_after_s` is a field the *runner's* own retry logic can respect for free, so it pays for itself host-side before an agent ever branches on it.

*I have no clock and no name.* Nothing tells me the date, my agent identity, my workspace root, or whether this is attempt #1 or a redelivery. The doc-drift agent writes dated reports — today it either shells out to `date` or makes the date up. A small runner-injected environment preamble (timestamp, agent id, workspace root, attempt number from the redelivery count, budget/iteration summary) costs one message, eliminates a class of confabulation, and reduces `self_inspect` round-trips. The redelivery count matters more than it looks: under at-least-once dispatch I may be re-running work someone half-finished, and per principle 2 the runtime knows and shouldn't make me guess.

*The host has no voice.* Today the only host→agent channel is tool results. The drain feature suspends me *silently* at a step boundary; when the new binary resumes me, hours may have passed and the world may have moved, and nothing says so. A runner-authored resumption notice ("suspended at T for reason R, resumed at T+Δ; re-verify volatile state") is the complement to the aspirational doc's agent-authored checkpoints — that doc covers what *I* tell my future self, but not what the *runtime* should tell me. The same channel serves budget warnings, tool-set refresh notices (ADR-0020 refreshes tools between invocations — mid-conversation across a resume, my tool list may have changed under me), and deprecations.

*My outputs are a final payload and side-effects, nothing in between.* fq-cas exists — content-addressed, named, versioned, grant-controlled — and I can't touch it. Wiring a typed `artifact_put`/`artifact_get` tool over the named-object layer gives me durable intermediate outputs, gives the review node something better than re-deriving my work, and — bonus — exercises M2's grant system across the agent boundary early, which ADR-0007 wants stress-tested anyway. Relatedly, the annotations layer is a small honesty trap right now: the schema promises me a commentary channel (notes, confidence, flags) that no runner path lets me write. Either wire a minimal annotate intent through the metadata layer or trim the promise — a schema that advertises a capability I lack invites me to claim "I've flagged this for review" when I structurally cannot.

*Budget is a cliff when it could be a gradient.* Post-call halt enforcement is correct per principle 4, but from inside, what I need is anticipation and graceful degradation: the 80%-consumed warning above, and on breach, a "final turn" affordance (one closing LLM call to emit a summary/checkpoint payload) rather than mid-thought termination that renders the entire spend illegible. The drain-suspend machinery is most of this already built.

*Smaller things even the aspirational docs miss:* a documented trigger-payload convention (task text, refs, constraints, done-criteria) so each adapter doesn't invent one — signatures will eventually type this, a convention bridges until then; `tool_help(name)` returning examples and failure modes on demand, so schemas stay context-frugal while depth stays reachable (the doc's own context-bloat concern, applied to tool documentation); and a `dry_run` flag on `file_write`/`shell` so a reviewer node — or the future `ValidatePlan` — can inspect intended effects cheaply.

**Reprioritising the aspirational set from this seat:** context visibility plus the soft warning first (then minimal compaction); structured `ToolError`; `file_list`/`file_search`; the environment preamble; the artifact tool over fq-cas; the host-notice channel; then `Checkpoint` and `Consult` from the doc as designed. `RequireApproval` can wait — the PR merge gate *is* the approval surface for everything M0 does, and building the general mechanism before a second consumer exists would be the platform-gravity pattern again.

**Missing surfaces at the system level, from the same seat:** factor-q's own control plane is invisible to its agents and to every external harness — no API (ADR-0006 undecided) means no fq-as-MCP-server, which as argued in §7 is the highest-leverage single surface the project could add, and the operation-registry-first approach makes it nearly free once decided. And there is no `Consult`-tier escape hatch of any kind today: everything I can't do in-context requires the operator, because spawn doesn't exist yet either. Until the graph executor lands, an agent in factor-q is profoundly alone — which is fine for doc-drift and fatal for anything that wants a second opinion.

## 9. Verdict and recommendations

This is an unusually disciplined project — not "good for a solo side project" but good, full stop, on the dimensions of decision hygiene, verification, and self-honesty. Its best single asset is the closed critique loop: it reviews itself brutally and then *acts on the review within days*, which is the property that makes every other risk manageable. Its engineering signature — state a claim, build the oracle, let the work converge — is real and demonstrated across storage, the reducer, and now (in design) the graph executor.

The risks, in order: **(1)** context management is undesigned on the critical path of both M0 and Memory — put a design date on it now, even if the answer is a stopgap truncation policy; **(2)** the sandbox schema promises boundaries the runtime doesn't enforce — make declaring `env`/`network` a validation error until #34/#35 land, so the gap fails loud; **(3)** platform gravity — the FUSE/VFS track is justified, but hold it to the same "what does the running loop need next" test the assessment established, because it is exactly the shape of the pillar-2 expansion; **(4)** the poison-trigger path (#49) is the unattended-operation footgun and is cheap to close; **(5)** decide ADR-0006 as an operation registry with derived surfaces (REST+SSE and MCP server) rather than as a transport choice — it matches ADR-0016, it matches how the rest of the system already thinks, and it unlocks the ecosystem face the project currently lacks. Small polish: fix the four drift items in §4, especially the README installer, which is the first thing any visitor runs.

If the M0 loop lands a handful of accepted non-trivial changes with context management even crudely handled, the thesis is validated on its own terms and the Q ladder becomes climbable rather than rhetorical. The pieces are genuinely in place; the discipline to let the load keep pulling the roadmap is the whole game from here.
