# factor-q code review — full synthesis

*Reviewed 2026-07-14 against the working tree @ `c9da957`. Produced by an eight-agent parallel cold review (Claude Code); findings marked ✓ were re-verified by hand against the source.*

**Scope:** working tree at `c9da957`, ~63k lines across `services/` (Rust), `adapters/github-watcher` (Go), ops/scripts, and the docs set. Eight independent cold reviewers each covered a slice (worker/reducer, event/control plane, runtime surface, CLI+tools, store, test infrastructure, periphery, architecture/onboarding); I then personally re-verified the highest-severity claims against the source. Findings marked **✓** below are ones I confirmed by reading the code myself; the rest carry the reviewer's exact file:line and held up to caller cross-checks in their reports, but spot-check before acting on any single line number.

## Verdict

This is an unusually strong codebase — well above the bar for a project of this age, and every reviewer independently reached that conclusion. The architecture is deliberate and legible: the two-role runtime (control plane vs worker), the pure-reducer/host-loop split, the store's `Cid → ContentStore → NameIndex → Repository → GatedRepository` layering, the dashboard as a read-only client of typed views, and the Go adapter held behind two written wire contracts are all real, load-bearing boundaries. Documentation discipline is the best feature: nearly every module opens with a `//!` contract citing the ADR or incident that motivated it, and the test stack (trace oracle, seeded deterministic simulation, differential models, conformance suite run over the wire) is genuinely rare.

The debt is concentrated and specific rather than diffuse: **(1)** three god-files at the composition seams (`runner.rs` 6,351 lines, `fq-cli/main.rs` 4,411, `mcp.rs` 2,088) in an otherwise well-factored tree; **(2)** copy-paste in the async plumbing (consumer loops, shutdown joins, schema-versioning kits); **(3)** the three-disjoint-cargo-projects decomposition, whose hand-synchronized lockfiles are a standing hazard the code itself admits; and **(4)** doc drift concentrated in exactly the files a newcomer reads first, while the deep docs stay current. Along the way the review surfaced a handful of genuine correctness/security defects worth fixing regardless of the quality agenda.

## Answers to the review questions

**Are the right components defined and composed together?** Yes, at every level except the build. Crate boundaries are meaningful (fq-tools isolating the sandbox and built-ins is exactly right; fq-cli stays a thin formatter over `fq_runtime::views`; the dashboard duplicates no projection logic and consumes the typed tarpc contract so drift fails the compile; the Go watcher deliberately shares no code and reconstructs the wire shape in tests). Two composition-level criticisms: the **three separate cargo projects** conflate crash domains (justified) with lockfiles (not justified — see theme 3), and **"what processes form a running deployment and how do they talk"** is not written down in any single place — the answer (daemon + fq-cas + fq-dashboard + github-watcher, joined by NATS subjects and two localhost tarpc sockets) is scattered across STATUS.md, a plan doc's diagram, the wire contract, and Cargo.toml comments; ARCHITECTURE.md never mentions tarpc at all.

**Do modules have clean, documented interfaces?** Best-in-class for module docs; both library crates have curated, rationale-carrying `lib.rs` re-export surfaces rather than `pub mod` sprawl. The gaps are small: one event variant has three parallel names maintained by three exhaustive matches in two files (NATS subject `llm.response`, schema id `llm_response@1`, projection `event_type`, plus `Debug`-lowercased failure kinds diverging from the wire's snake_case); `test_support` is `#[cfg(test)]`-gated so the external integration crate can't reach it; and a few doc comments are misplaced or stale (details per component).

**Can a new contributor read it?** The user path (README → STATUS → QUICKSTART) is genuinely good; the contributor path has real breakage. CONTRIBUTING's canonical command **✓** `FQ_NATS_URL=nats://localhost:4222 just test` overrides the justfile's correct token-bearing default and fails auth against the bundled broker — a newcomer burns an hour on that. ARCHITECTURE.md's "implementation status" half is a milestone stale (misses the dashboard, read service, and ~6 CLI verbs), so the reader learns to distrust it and falls back to `lib.rs` docs — which are excellent, but that's a discovery, not a path. The other tripwires: navigating the three god-files, and comment jargon ("finding 5, slice-7 soak", "ADR-0018 §4") that is precise but assumes `docs/` is open in the next window.

**Consistent ways of doing things?** Strikingly consistent for a codebase this size: thiserror in libraries / anyhow in binaries, tracing only in binaries, deny-warnings mirrored in every manifest, `fq-*`/`FQ_*` naming, and — most notably — **one testing culture** (seeded, reproducible simulation with printed seeds) across two projects that share no code. Divergences worth converging: three schema-versioning approaches across the three stores sharing one SQLite file (worker store's versioned kit, control-plane store's copy of it, projection's ad-hoc `pragma_table_info` ALTERs); two NATS-gating idioms in tests; CLI error paths split between `anyhow` returns and `std::process::exit(1)`, with fatal errors routed through `tracing::error!` to **stdout** (polluting `--json | jq` pipelines); `--json` vs `--format {pretty,json}`.

**Idiomatic Rust/Go?** Largely yes — validated newtypes at serde boundaries, essentially no `unwrap` in production paths, adapters confined to single modules (rmcp in `mcp.rs`, genai in `llm/genai.rs`), textbook Go interfaces with a pure planner. Specific lapses: blocking `std::fs` in async paths (workspace provision/reclaim, discovery's `glob()` walk), hand-rolled entropy via hashing `SystemTime` (`runner.rs:3832`), `unsafe env::set_var` in parallel tests (a real data race, Rust 2024 made it unsafe for this reason), `tokio = { features = ["full"] }` in the two library crates while fq-store carefully trims to six features, and a hand-rolled channel poll-merge duplicated in two places that `tokio_stream::StreamMap` replaces.

## Cross-cutting themes

### 1. The tree is right; three files at the seams have accreted

The module *tree* is well-factored — the god-files sit precisely where subsystems compose, and each already has internal section banners marking the natural split lines, so the refactors are mechanical:

- `worker/reducer/runner.rs` (6,351 lines) — host loop + resume/replay + MCP sampling/elicitation/evaluator servicing + elicitation schema validation + ~2,450 lines of tests. Split the ADR-0018 server-request machinery and replay helpers into siblings; also bundle the invocation-scoped state into an `InvocationCtx` (`run_loop_inner` takes 19 parameters; the file has six `#[allow(clippy::too_many_arguments)]`).
- `fq-cli/src/main.rs` (4,411 lines, 14 subcommands, ~1,120-line `run_daemon` with eleven spawned tasks) — the `// ===` banners are doing a module tree's job.
- `mcp.rs` (2,088 lines, eight responsibilities) — the reviewer judged the internal boundaries already clean enough to split into `mcp/{handler,manager,tools,notifications,prompt_convert}.rs`.

### 2. Copy-paste concentrated in the async plumbing

Individually small, collectively a pattern — the highest-stakes one first:

- **✓-adjacent (correctness stakes):** `resume()` re-derives agent config, sandbox binding, and step-0 context by hand-copying the fresh-path code (`runner.rs:1083` vs `:738–824`); the resume/fresh observational-equivalence guarantee depends on copies never drifting. Extract one `build_invocation_setup`.
- The select-on-shutdown/deserialize/ack-or-NAK durable-consumer loop exists three times (`heartbeat_consumer.rs:74`, `projection/consumer.rs:60`, `coordination_consumer.rs:180`) plus two test variants.
- Three near-identical ~50-line control listeners (reload/drain/down, `main.rs:2210,2281,2340`) and eight identical shutdown join-with-timeout blocks (`main.rs:2631–2733`).
- The SQLite versioning kit (`Compatibility`/`check_compatibility`/`split_sql`) duplicated between `worker/store.rs` and `control_plane/store.rs` while the projection store has no versioning at all.
- Small fry: `require_nats` ×2, splitmix64 test RNG ×2, `human_bytes` ×2 (with conflicting KB/KiB conventions), poll-merge ×2, the MCP startup loop duplicated between `trigger_agent` and `run_daemon`, `parse_status` re-implementing `OwnerStatus::parse`, and the justfile's Cargo-version grep duplicated in `package.sh`.

### 3. One workspace instead of three cargo projects

The strongest architectural recommendation from the review. Separate *binaries and crash domains* are well-argued in the docs; separate *lockfiles* are not. Today fq-dashboard path-compiles fq-runtime under a second `Cargo.lock` and hand-mirrors transitive pins under a `LOCKSTEP` comment that admits it already broke once (rmcp 1.8); there is no `[workspace.dependencies]`, versions diverge across projects (axum 0.7 vs 0.8), and there is no single `cargo test`. Unifying the workspace preserves the binaries and crash domains, deletes the LOCKSTEP hazard, and subsumes the dependency-hygiene findings. Related toolchain skew: the Dockerfiles build with `rust:1.85` while `rust-toolchain.toml` pins 1.95.0 and the build context excludes the pin file — contradicting the pin's "same commit implies same binary" rationale.

### 4. Doc drift is concentrated in the first-read files

The deep docs (event schema, trigger wire contract, module docs, ADRs) are current. The shallow docs — the ones a newcomer reads first — are one milestone behind, and two of them are wrong in security-relevant ways:

- **✓ high** SECURITY.md:15 and STATUS.md claim the bundled NATS has "no authentication"; `infrastructure/nats/nats.conf` enforces token auth (and its own comment says auth is mandatory). SECURITY.md calls itself canonical — update it first.
- **✓ high** CONTRIBUTING.md:24,66 — the canonical test command fails against the bundled broker (tokenless URL overriding the token-bearing justfile default). Also ~4× stale test counts, and the "skips without NATS" claim is no longer true since the justfile always exports a default URL.
- **✓ med** `just ci` (justfile:113) — self-described "full local gate" — omits the dashboard phases that GitHub CI requires, and the root `just build`/`just test` ("all services") only delegate to fq-runtime, silently skipping store and dashboard. Both break AGENTS.md's local/CI-isomorphism promise.
- ARCHITECTURE.md's implementation half: CLI table lists 8 of ~14 commands; the shipped read-service/views/tarpc/dashboard subsystem is absent entirely. README and CONTRIBUTING project trees list a nonexistent `skills/` directory and omit `fq-dashboard`, `adapters/`, `ops/`. QUICKSTART.md:88's example subject `fq.agent.>.tool.call` is an invalid NATS filter (`>` is only legal as the final token). The dashboard's module doc still claims "zero client-side JS" while the file vendors datastar.js and serves SSE.

### 5. The test harness is a jewel with a discoverability problem

The 6k-line `test_support/` is a layered verification architecture, not an accretion — but its `mod.rs` index documents only two of six modules (the four a newcomer needs most — `oracle`, `sim`, `runtime`, `mock_anthropic` — are unlisted), the crown-jewel DST suites (~2.4k lines: fault sweeps, resume-equivalence, soak) hide *inside* `test_support/sim.rs` rather than near `worker/reducer/`, and the `#[cfg(test)]` gate forces `tests/mcp_integration.rs` to triplicate its runner wiring. Exposing it behind a `test-support` cargo feature (as fq-store does with `conformance`) fixes the duplication and lets the rustdoc examples compile.

## Per-component notes

**Worker/reducer (`worker/`, 12.3k)** — the strongest component. The reducer/host split delivers suspension/replay/migration structurally; WAL discipline is exemplary (guarded transitions, refuse-if-newer migrations, and the standout meta-test `wal_tables_are_keyed_by_invocation_id` that enumerates `sqlite_master` so new tables can't break the keying invariant). Beyond the table above: `store.rs:31`'s "not yet done" doc lists three things that shipped long ago; `DurableStart::set_attempt` is dead code; `run_migrations` is six copy-pasted version blocks that a `const MIGRATIONS` table replaces; a test leaks temp dirs via `std::mem::forget(store_dir)` (`runner.rs:4826`); `FailurePhase::LlmResponse` is used as a catch-all for three non-LLM failure modes.

**Event/control plane (6.1k + top-level event modules)** — coherent event-sourcing: three-layer schema with a type-level consumer barrier, idempotency designed end-to-end with redelivery tests, and the dispatcher's ack-on-durable-start work encodes the actual 2026-07-06 redelivery-storm incident as regression tests. Besides the table: `ensure_event_stream` runs `update_stream` on **every** connect, so any short-lived CLI invocation rewrites broker stream config (`bus.rs:305`); dead `is_due`/`retention_cutoff_ms` re-implemented in `retention.rs`; `parse_llm_response` returns a tuple element that is `None` on both branches, making a downstream `or()` a no-op.

**Runtime surface (mcp/config/agent/llm/…, ~12k)** — contract-first docs, owned provider-neutral types, validated newtypes (`AgentId` checks in its serde `Deserialize`). Besides the table: `to_snapshot`'s "captures all fields" doc/test overstate (four fields omitted); per-server `env` goes through a `HashMap` so declaration order is nondeterministic; `split_frontmatter` accepts a `---` inside a YAML string as the closing fence; genai's `map_error` never yields `RateLimited` so 429s are indistinguishable in telemetry.

**CLI + fq-tools (8.4k)** — no high-severity bugs; the sandbox module is exemplary (canonicalisation containment, documented TOCTOU limits, the `MisquotedPath` self-diagnosis for LLM-mangled paths is genuinely smart) and SIGPIPE/SIGTERM handling is correct and kernel-level tested. Besides the god-file: `fq workers --help` documents `prune` with `show`'s text because a doc comment sits on the wrong variant (`main.rs:274`); `drain_daemon`'s doc is a copy-paste of reload's; fatal errors go to stdout via tracing and vanish under `RUST_LOG=off` (`main.rs:460`); a 30 s stale threshold is hard-coded in four places while `DEFAULT_STALE_THRESHOLD_MS` is `pub`; discovery's `glob()` collects every match before applying the cap, blocking the runtime on `**/*`; smoke-test `run_fq` reads piped output only after exit — a latent 64 KB pipe-buffer deadlock.

**fq-store (9.2k)** — spec-grade docs (named invariants I1–I5/A1–A6 traced to design docs and TLA⁺) and the best test suite in the repo (seeded differential sims with per-step invariant oracle, conformance suite reused in-process and over tarpc). The four seam-level items are in the table. Smaller: `NameIndex` mixes five user-facing ops with nine GC-protocol internals (split a sub-trait); crash-orphaned `.tmp.*` staging files are invisible to the audit's reaper and accumulate forever; authorization denials emit no tracing (an access-control boundary should log denials); `Cid::from_hex` maps a CLI typo to "store corruption"; `recompute_liveness` is quadratic over the log's life; the `name < "{prefix}/"` bound silently depends on `/` being the ASCII successor of `.` — one comment away from someone "fixing" it.

**Test infrastructure** — covered in theme 5. Additional: the smoke suite's sandbox-denial test asserts only that the run completed, never that anything was denied (`smoke.sh:297`) — the one real assertion gap found; a few fixed sleeps (200 ms consumer registration, 500 ms express warmup, `sleep 2` projection catch-up) where sibling code already shows the poll-with-deadline pattern; panicking assert helpers lack `#[track_caller]`; skip-by-early-return reports gated tests as *passed*; `health.rs` is the only src module with zero tests.

**Periphery (dashboard, Go watcher, ops)** — unusually disciplined; the deploy script is rigorous ops shell (checksum + embedded-SHA cross-verification, escalating shutdown, `/proc/<pid>/exe` verification, atomic symlink flip). Besides the table: `find_browser`'s fallback dies under `set -euo pipefail` before its friendly error prints; `pgrep -x fq` can target an operator's concurrent CLI process; deploy relaunches daemon and watcher but not the dashboard over an unversioned bincode wire; `install.sh` silently skips checksum verification when the `.sha256` fetch fails; the Go watcher never reads `envelope.schema_version` — the exact drift class behind its own past `payload.payload` bug; `envIntOr` silently swallows malformed env values ten lines below strict flag validation.

## What I'd do first

1. Fix the four **✓** quick-win bugs: MCP dedup key (include `url`/`env` or key on name), CONTRIBUTING's test command, SECURITY/STATUS auth claims, `just ci`/`just test` coverage gaps. All are small diffs.
2. Design-level look at the store seams (#2, #4, #5): gate scope encoding, manifest claim protocol, raw-put/GC composition — these gate anything built on fq-store, and the crate's own invariant vocabulary (A2, S1) already names what's violated.
3. Add a WAL sequence column and shared `build_invocation_setup` for resume (#3 + the fresh/resume duplication) — both protect the same replay-equivalence guarantee.
4. Projection retention + heartbeat filtering (#6) before dogfood data grows into it.
5. Unify the cargo workspace (theme 3) — kills LOCKSTEP, version skew, and the "which lockfile" class permanently; bump the Dockerfile toolchains in the same pass.
6. Split the three god-files along their existing banners; extract the durable-consumer helper and the schema-versioning kit while in there (themes 1–2).
7. Un-gate `test_support` behind a cargo feature and rewrite its `mod.rs` as a "which harness do I want" map (theme 5).
8. Refresh ARCHITECTURE.md's implementation half and add the missing one-page process/IPC deployment diagram — the single question no document currently answers.

Housekeeping aside: six worktrees linger under `.claude/worktrees/` (ci-caching, ci-timing, exec-timeout, feat-92-shell-teaching-errors, hermetic-deploys, operator-dashboard) — worth a sweep to confirm none hold unmerged work.
